# 设计比稿子流程

当用户想先出多版方向、做比稿、比较 A/B/C 方案，再决定推进哪一版时，走这里。

## 目标

围绕同一个业务问题，产出多组可比较的设计方向，而不是只给单一方案。

本子流程只定义分案策略、差异化约束和交付格式，不规定某个特定执行工具。

如果用户希望这些方向能在页面里直接切换或对比，还要额外补一层 tweak 暴露，让比稿结果成为可编辑属性，而不只是静态说明。

## 方案数量

- 用户明确指定数量时，严格按用户要求
- 用户没有指定时，默认产出 `3` 个方向

默认三案可用这组梯度：

- `A 稳健型`
- `B 平衡型`
- `C 突破型`

如果用户自己给了命名或分档方式，优先使用用户版本。

## 先锁共同约束，再拉开差异

先整理这些共同约束：

- 业务目标与成功标准
- 页面或组件范围
- 用户已明确锁定的要求
- 不能改的内容

已锁定的要求必须在所有方案里保持一致，不能拿来充当差异点。

## 差异化框架

差异优先从这三层里挑 3-6 个未锁死维度来拉开。

### 1. 骨架层

决定界面的物理形态与任务动线，例如：

- 页面 / 抽屉 / 弹层 / 分栏
- 线性引导 / 模块枢纽 / 并行操作

### 2. 信息与交互层

决定内容组织、密度与操作方式，例如：

- 高密度铺陈 / 重点收纳 / 极简聚焦
- 表单点击 / 快捷操作 / 直接操纵

### 3. 视觉与反馈层

决定气质和细节手感，例如：

- 系统原生 / 现代卡片 / 更有记忆点的表达
- 克制反馈 / 顺滑过渡 / 更强存在感的反馈

注意：

- 差异不能只是换色、换皮、换插画
- 至少要在骨架层或信息与交互层拉开真实差异
- 视觉层更适合增强差异，不适合单独承担全部差异

## 生成方式

推荐顺序：

1. 先做一轮规划，明确锁定项、可变化维度、方案矩阵
2. 再分别展开每个方向
3. 最后做汇总对比与推荐

不要把多个方案写成一个含混的大杂烩。每个方向都要有清楚的定位和边界。

要以“给用户惊喜”为目标，充分发挥前端和 AI 的能力，而不是只做安全但平的方案。

如果当前环境支持子代理，优先把不同方向拆成独立任务并行推进；主代理负责统一约束、最终对比和推荐。

## 比稿属性暴露

比稿不只是“生成几版说明”。只要用户希望在页面内切换方案，或当前实现本身适合通过属性切换方向，就要把方案做成 tweak。

优先做法：

- 使用 `@axhub/genie-editor-client`
- 通过宿主的 `schema / values / adapter / update` 暴露方案选择
- 方案字段优先使用 `card` 类型，而不是普通下拉

`card` 适合表达“多方向里选其一”的比稿切换。一般每个选项至少包含：

- 标题
- 描述
- 对应的稳定 `value`

写法上，优先让每个方案对应 `options[]` 里的一个卡片项：

- `label` 放方案标题
- `description` 放一句简短说明
- `value` 放宿主回写时使用的稳定标识

默认建议：

- 用户明确指定方案标题和描述时，按用户要求
- 用户没指定时，为每个方向补一条足够区分的标题和一句描述
- 卡片描述要强调该方案的核心差异，不要写空泛口号

如果当前页面只是先做方向探索、还没有合适的宿主承接点，可以先完成比稿内容本身；一旦用户要求“能切换看”“放到调整面板里选”，就把这一步升级为属性编辑任务。

## 交付格式

最终至少给出：

1. 方案列表
2. 每个方向一句话概述
3. 一张简洁的差异表
4. 推荐优先推进方向

如果本轮同时落了页面内切换能力，还应补充：

5. 对应的 tweak 字段设计
6. 每个 `card` 选项的标题、描述、value

差异表至少覆盖：

- 骨架层差异
- 信息 / 交互层差异
- 视觉 / 反馈层差异
- 适合的业务取向或风险取向

## 质量检查

- 所有方案是否都围绕同一个业务目标
- 是否严格遵守用户指定数量；若未指定，是否回落到默认 3 案
- 差异是否来自可变化维度，而不是违背锁定约束
- 是否至少在结构或交互层存在实质差异
- 最终是否明确给出推荐推进方向
- 如果需要页面内切换，是否已经通过 `@axhub/genie-editor-client` 暴露为 `card` 类型 tweak
- `card` 选项是否都有可读标题和简洁描述
