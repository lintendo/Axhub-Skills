# 属性编辑子流程

当用户要暴露 tweak / 调整面板字段、整理可编辑属性、补页面级属性聚合，或让某个元素“能在调整面板改”时，走这里。

## 目标

为目标元素或页面现有 tweak-capable 节点补齐可编辑的 `schema / values / adapter / update` 能力，让 Genie Editor 能读取并回写这些属性。

这不是把组件全部 props 原样摊开，而是把真正有用、可读、可维护的编辑能力暴露出来。

## 基本口径

- 用户明确指定要暴露哪些属性时，优先按用户要求
- 用户没有明确指定时，按默认暴露原则生成；一个组件或一个页面默认暴露 `5~8` 个对 UI/UX 影响最大的属性
- 宿主负责业务字段、默认值和更新语义
- 编辑器只消费 `schema / values / update` 结果，不替宿主决定业务字段
- Genie 后端不在线时，也可以直接在当前项目中完成属性实现

需要快速确认字段类型和推荐场景时，读 `references/schema-guide.md`。

## 默认暴露原则

### 1. 先看视觉感知，再看底层 API

优先暴露最影响第一视觉和组件骨架的字段，例如：

- `size`
- `variant`
- `color` / `theme`
- 需要长期配置的标题或核心文案
- 核心状态，如 `disabled`、`error`

低频、技术味很重、用户通常不会主动调的字段，默认不要直接摊开。必要时折叠到高级设置，或由 agent 结合上下文代管。

### 2. 文本字段只在需要配置时暴露

标题、文案、副标题、按钮文案这类内容，本身可直接文本编辑；只有在需要长期配置、统一整理，或明确要求放进调整面板时，再做成 tweak。

### 3. 第三方库优先做封装，不做透传

如果底层依赖第三方库，不要让用户直接面对生硬的技术字段名。

做法：

- 先把用户自然语言需求翻译成宿主侧 schema 字段
- 再由宿主的 `update` 逻辑映射到第三方库具体 prop
- 除非用户明确要求，否则不要把底层 prop 名当成主要暴露面

### 4. 优先暴露语义化 token，而不是绝对值

为了让结果更稳、更现代：

- 间距、圆角、尺寸优先用 `sm / md / lg / full` 这类枚举
- 颜色优先用主题 token、业务语义色或已存在的设计系统变量
- 只有在用户明确要求，或业务上确实需要时，才开放自由输入的绝对值

### 5. 复杂数据组件要看数据形态

图表、表格、复杂展示组件，不是字段越多越好。先看数据本身，再决定暴露什么。

例如：

- 单线图没必要默认暴露图例开关
- 多系列数据才值得暴露分类颜色、图例位置、系列样式
- 数据结构没支持的能力，不要硬做成可调项

## 常见落点

### 1. 单元素属性编辑

适用于：

- 某个卡片、按钮、图表、模块要补 tweak
- 用户已经选中了具体元素
- 目标是让当前元素可调
- 用户希望标题、文案等文本也能在调整面板中配置

实现重点：

- 给目标元素补 `schema`
- 提供当前 `values`
- 通过 `adapter` 绑定到元素
- 让 `update` 能把 patch 正确回写到宿主状态

### 2. 页面级属性聚合

适用于：

- 页面里已经有多个 tweak-capable 节点
- 用户要按页面统一查看或整理属性
- 用户关注的是页面聚合后的调整体验

实现重点：

- 优先复用已有元素级 tweak 注册
- 按节点聚合，不把页面级能力写成一坨全局静态配置
- 页面聚合只改变展示与组织方式，不改变元素级 owner 关系

## 实施提醒

- 术语尽量对齐 `schema`、`values`、`adapter`、`update`
- 优先复用现有 tweak protocol 和宿主状态，不另造一套平行配置体系
- 字段结构发生变化但没有重建注册时，记得考虑刷新 editor 对 tweak 列表的可见性
- 如果当前环境支持子代理，可把具体组件实现拆出去；主代理仍保留整体约束与最终复核
